Or alternatively, ensure the guards are evaluated with the correct data by checking the passed data directly rather than relying on self.data:

```python
def trigger(self, event: str, data: Optional[Dict] = None) -> bool:
    if data:
        self.data.update(data)
    
    # Check passed data for success/failure rather than self.data
    success = data.get("success") if data else None
    
    possible_transitions = self.transitions.get(self.current_state, [])
    
    for to_state, guard_fn, description in possible_transitions:
        if guard_fn(self):
            return self.transition(to_state)
```

No wait, the guards are lambdas that close over `sm`, not `data`.

OK I think the real fix is simpler: **The bug is that the guards use `not` which inverts the logic incorrectly for boolean False. When success=False, `not False` is True, but what we really want is to check if success is exactly the boolean False, not just any falsy value.**

The fix is to use explicit `is False`:

```python
("FAILED", lambda sm: sm.data.get("success") is False, "failure")
```

This ensures we only go to FAILED when success is explicitly set to False, not when it's any falsy value like None or missing.
