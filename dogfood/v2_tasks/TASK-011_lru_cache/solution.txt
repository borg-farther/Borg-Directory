## Solution for LRU Cache Bug

### The Bug

In the `get()` method, `move_to_end()` is called BEFORE returning the value. This is correct behavior. However, the bug is actually in the **interaction between get() and put()** — specifically that `get()` does NOT check if the cache is at capacity before doing `move_to_end()`.

Wait, let me re-analyze. The actual bug is:

In `put()`, the eviction check happens BEFORE adding the new item:
```python
if len(self.cache) >= self.capacity:
    evicted_key = next(iter(self.cache))
    del self.cache[evicted_key]
```

This is correct! The issue is that after `get("a")` is called, "a" moves to the end. So the first item (LRU) is now "b". When we add "d", we evict "b". That seems right...

Let me trace through more carefully:

After `put("a")` → cache: OrderedDict([('a', 1)]) — end=a
After `put("b")` → cache: OrderedDict([('a', 1), ('b', 2)]) — end=b
After `put("c")` → cache: OrderedDict([('a', 1), ('b', 2), ('c', 3)]) — end=c
After `get("a")` → cache.move_to_end("a") → OrderedDict([('b', 2), ('c', 3), ('a', 1)]) — end=a

Now `put("d")`:
- len(cache) = 3, capacity = 3, 3 >= 3 is True
- We evict: next(iter(cache)) = 'b'
- After del: OrderedDict([('c', 3), ('a', 1)])
- After cache['d'] = 4: OrderedDict([('c', 3), ('a', 1), ('d', 4)])

This gives us ['c', 'a', 'd'] which is CORRECT!
